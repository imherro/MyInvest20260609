# 功能清单

## 顶层目标

建设一个**本地大模型辅助的投资研究与风控系统**，用于保存证据、结构化研究快照、决策记录、组合偏离、影子账户历史和复盘材料。

本系统的首要目标不是替代人工交易，而是：

1. 让每日研究有证据、有留痕、有结构化数据。
2. 让组合暴露、仓位偏离和风险变化可追踪。
3. 让 AI 结论经过校验、审核和人工确认后才进入最终历史库。
4. 让真实账户永远保持只读导入，绝不自动交易。
5. 让影子账户只做纸面模拟，用于验证策略和对比真实账户。

## 最高边界

本系统不生成真实交易指令，不连接真实下单接口，不自动买卖。

所有输出只能是：

- 研究记录
- 风险提示
- 仓位区间
- 比例偏离
- ResearchFirst 清单
- 影子账户纸面模拟
- 人工可审查的候选建议

真实交易只能由人工在券商客户端中独立完成。

## 核心架构原则：JSON First，Markdown on Demand

系统采用 **JSON First，Markdown on Demand**。

AI 日常研究模块只允许直接生成结构化 JSON，不直接生成 Markdown 报告。JSON 是唯一事实源，用于：

- 入库
- 审计
- 复盘
- 曲线展示
- Web 工作台
- ChatGPT 审核
- 历史对比
- 自动测试

Markdown / HTML / PDF 是派生视图，由脚本基于 JSON 或 SQLite 按模板渲染，用于人工阅读、导出和归档。

推荐流程：

```text
AI 生成 JSON → JSON Schema 校验 → SQLite 入库 → Web 展示 → 按需渲染 Markdown / HTML / PDF
```

禁止流程：

```text
AI 写 Markdown → 从 Markdown 解析 JSON → 入库
```

理由：

1. 节省 token，AI 不用同时写 JSON 和长篇 Markdown。
2. 避免 Markdown 与 JSON 结论不一致。
3. JSON 可 schema 校验，便于 blocked / finalized 状态控制。
4. Web 工作台天然读取 JSON / SQLite，不需要 Markdown 中间层。
5. 报告格式由模板统一，不随 AI 文风漂移。

## 角色分工

| 角色 | 能做 | 不能做 |
|---|---|---|
| 脚本 | 数据采集、清洗、校验、入库、画图、比对、安全扫描、报告渲染 | 不解释市场，不新增研究结论 |
| Codex | 生成研究 JSON、整理证据、生成审核包、实现代码和测试 | 不能最终定稿，不能影响真实账户，不能绕过安全检查，不能直接写最终 Markdown |
| ChatGPT 网页版 | 复核数据缺口、逻辑冲突、风险遗漏、结论强度 | 不直接写入最终库，不直接改变真实账户 |
| 人工 | 确认最终报告、调整风险边界、维护标的池、决定真实操作 | 不得省略审核与留痕流程 |

最终入库流程：

```text
Codex JSON 初稿 → JSON 校验 → 安全检查 → ChatGPT 复核 → 人工确认 → finalized 入库
```

## 核心功能

### 1. 数据源

- Tushare：A 股结构化主源。
- QMT：盘中价格和真实持仓只读导入。
- BaoStock：A 股交叉验证和备用数据源。
- FRED / yfinance / 网络：宏观、海外和补充证据。

数据源要求：

1. 每次采集必须记录来源、数据日期、采集时间和版本。
2. 数据缺失、过期或冲突时，必须写入 `data_gaps` / `conflicts`。
3. 超过冲突阈值的模块不得进入最终报告。
4. QMT 只读导入失败时，保留上一快照并提示，不得伪造最新持仓。

### 2. 标的池

标的池由以下部分组成：

- 主线 ETF
- 主线龙头
- 防御 ETF
- 现金 ETF / 短融 ETF
- 真实持仓
- 观察标的
- ResearchFirst 标的

规则：

1. 真实持仓从 QMT 只读导入后自动进入标的池。
2. 新增池内标的必须有来源、分类、加入原因和复查日期。
3. 未完成研究的标的进入 ResearchFirst。
4. ResearchFirst 标的不得进入影子账户。
5. ResearchFirst 标的不得进入 `action_candidates`。
6. ResearchFirst 只能输出“需要补哪些研究”，不能输出买卖建议。
7. 标的移出池子时必须保留历史原因、移出日期和操作人。

### 3. 研究历史

市场、主题、ETF、个股、组合和估值研究的 AI 原始输出必须是**可校验 JSON 快照**。

JSON 快照必须保存：

- 数据基准日期
- 生成时间
- 模块名称
- 数据来源
- 数据缺口
- 数据冲突
- 关键事实
- 结论摘要
- 判断理由
- 风险
- 结论强度
- 可行动性
- 失效条件
- 下次复查日期
- 明确禁止事项
- 当前状态

Markdown / HTML / PDF 不作为主数据源。需要人类阅读版时，通过 `render_report.py` 从 JSON 或 SQLite 渲染。

### 4. 自动研究节奏

首版不追求每天让 AI 重写全部观点。系统采用：

| 类型 | 频率 | 内容 |
|---|---|---|
| 日频事实 | 每个交易日 | 数据采集、数据校验、市场仓位轻量评分、真实持仓比例偏离、风险提醒、净值更新 |
| 周频观点 | 每周 1-2 次 | 主线强弱、主题拥挤度、ETF 估值、组合大调整候选、龙头排序 |
| 触发式研究 | 按事件 | 新标的加入、大涨大跌、价格偏离合理区间、重大新闻、财报、政策变化、人工要求 |

禁止让 AI 每个交易日无证据地大幅改变主线观点或组合建议。

### 5. ChatGPT 审核定稿

Codex 负责生成 JSON 审核包，提交给网页版 ChatGPT 复核。

ChatGPT 重点检查：

- 数据缺口
- 数据冲突
- 逻辑冲突
- 风险遗漏
- 结论强度是否过高
- 是否存在 ResearchFirst 越权
- 是否泄露金额、股数、账户号或真实订单信息
- 是否把影子账户误写成真实交易建议
- JSON 字段之间是否自相矛盾

ChatGPT 复核后进入 `chatgpt_reviewed`，再由人工确认后进入 `finalized`。

### 6. 报告渲染

AI 日常研究只生成结构化 JSON，节省 token 并避免 Markdown 与 JSON 双份结论不一致。

需要 Markdown / HTML / PDF 时，由脚本按模板渲染：

```bash
python scripts/render_report.py \
  --input research/finalized/2026-06-15_market_position.json \
  --template templates/market_position.md.j2 \
  --output reports/2026-06-15_market_position.md
```

渲染报告只能读取 JSON 或 SQLite，不能自行新增事实、修改结论、改变结论强度或改变可行动性。

渲染产物应记录来源 JSON 的 `snapshot_id`、文件名或 hash，保证可追溯。

### 7. 市场仓位

每个交易日生成一次市场仓位 JSON 快照。

保存内容：

- 建议权益仓位区间
- 市场评分
- 风险等级
- 拥挤度惩罚
- 数据日期
- 生成时间
- 证据列表
- 失效条件
- 结论强度
- 可行动性

市场仓位模块只决定“能打多大仗”，不直接决定买哪个标的。

### 8. 标的估值

记录每次股票或 ETF 研究。

保存内容：

- 合理价格区间
- 评级
- 置信度
- 估值方法
- 当前价格
- 数据缺口
- 失效条件
- 下次复查日期

估值频率建议：

| 类型 | 默认频率 |
|---|---|
| 宽基 ETF | 每周一次 |
| 行业 / 主题 ETF | 每周 1-2 次 |
| 个股 | 触发式 |
| 真实重仓 | 每日轻量检查 |
| 新加入标的 | 必须先完整研究 |
| 价格偏离合理区间 10% 以上 | 触发复查 |
| 重大新闻 / 财报 / 政策 | 触发复查 |

禁止每天全量重估所有标的，以免制造噪音和结论漂移。

### 9. 组合分析

组合分析只使用：

- 比例
- 百分点
- 目标区间
- 风险桶
- 净值指数
- 回撤
- 主题 / 行业暴露

组合分析不得展示：

- 总资产
- 持仓金额
- 股数
- 收益金额
- 真实订单
- 成交明细
- 账户号

默认组合基准模板：

```yaml
portfolio_policy:
  core:
    name: 宽基 / 核心底仓
    target: 40
    min: 30
    max: 50
  growth:
    name: 进攻 / 主线成长
    target: 40
    min: 25
    max: 50
  defensive:
    name: 防御 / 现金 / 红利 / 低波
    target: 20
    min: 10
    max: 35
```

组合分析优先回答：

1. 当前比例偏离目标区间多少？
2. 市场仓位是否要求整体升档或降档？
3. 哪些标的需要 ResearchFirst，而不是直接换仓？
4. 哪些候选只是 `rebalance_candidate`，不能写成交易指令？

### 10. 持仓对比

从 QMT 只读导入真实持仓快照。

支持对比：

- 真实持仓净值指数
- 影子账户净值指数
- 主要指数基准
- 回撤
- 换手率
- 行业 / 主题暴露
- 风险桶偏离

对外展示统一使用比例和指数化曲线，不展示真实金额和股数。

### 11. 影子账户

影子账户是纸面模拟账户，仅用于验证模型策略和对比真实账户。

首版影子账户采用固定规则，不让 AI 自由调仓。

默认规则：

```text
目标组合 = 宽基 40% + 进攻 40% + 防御 20%
单一 ETF 上限 = 15%
单一个股上限 = 5%
ResearchFirst = 0%
现金 ETF / 短融 ETF 下限 = 5%
权益总仓位必须落在 market_equity_min ~ market_equity_max 内
偏离超过 3 个百分点才模拟调仓
```

其他规则：

1. 只做纸面模拟，不发送真实订单。
2. 持仓只能来自标的池。
3. 池外标的无法被影子账户买入。
4. ResearchFirst 标的权重必须为 0。
5. 默认使用收盘价估算成交。
6. 记录手续费、滑点假设、换手率、回撤和基准收益。
7. 审核失败或数据阻断时，保持上一日持仓或进入现金规则。

### 12. 决策日志与结论漂移检测

决策日志采用追加式记录。

保存内容：

- 旧结论
- 新结论
- 证据
- 原因
- 影响
- 失效条件
- 下次复查日期
- 结论强度
- 是否重大变化
- 是否需要人工复核

如果结论发生重大变化但没有新增证据，系统必须提示：

```text
结论发生重大漂移，但缺少新增证据。
```

建议新增漂移检测字段：

```text
module
subject
old_conclusion
new_conclusion
change_reason
changed_by
evidence_added
market_data_changed
is_material_change
requires_human_review
```

### 13. Web 工作台

Web 工作台首版只做只读展示，不提供真实交易入口。

首版页面：

1. 今日状态页
   - 数据是否成功
   - QMT 是否导入
   - 今日报告状态
   - 是否 blocked

2. 仓位页
   - 建议权益仓位区间
   - 实际权益仓位
   - 偏离百分点

3. 标的池页
   - 已研究
   - ResearchFirst
   - blocked
   - next_review_date

4. 影子账户页
   - 净值曲线
   - 回撤
   - 换手率
   - 与指数对比

5. 审核包页
   - 今日待审
   - 数据缺口
   - 逻辑冲突
   - 风险遗漏

默认不做后台交易、不做账户金额展示、不做真实订单展示。

### 14. 安全检查

安全检查必须是独立 gate，不依赖 AI 自觉。

禁止进入 GitHub、最终 JSON、渲染报告或对外页面的内容：

```text
.env
*.db
qmt_export/*
real_positions/*
account_id
total_asset
cash_balance
position_amount
share_count
order_id
trade_id
真实账户号
总资产
持仓金额
股数
真实订单
成交明细
```

安全检查失败时，状态必须进入 `blocked`，不得生成最终报告。

## 首期范围

首期拆成 P0a / P0b / P0c，避免一次性做成过大的自动投研平台。

### P0a：证据库和审核闭环

目标：连续 5 个交易日稳定跑通“数据 → JSON → JSON Schema 校验 → SQLite / Web → 审核包”。

包含：

1. SQLite 表结构。
2. JSON schema 和校验器。
3. 安全扫描器。
4. 标的池基础表。
5. QMT 只读持仓导入 mock / adapter。
6. JSON 入库器。
7. 报告渲染脚本 `render_report.py`。
8. 市场仓位快照模块。
9. 审核包生成器。
10. 状态流：draft → json_validated → review_package_created → chatgpt_reviewed → human_approved → finalized。

不包含：

- 复杂主线识别
- 龙头排序
- 自动复盘评分
- AI 自由调仓
- 真实交易接口

### P0b：影子账户和曲线对比

在 P0a 连续稳定后加入：

1. 影子账户持仓表。
2. 影子账户纸面交易记录。
3. 影子账户净值曲线。
4. 回撤和换手率。
5. 与真实账户、主要指数的同口径对比。
6. 池外标的禁止买入测试。
7. ResearchFirst 权重为 0 的测试。

### P0c：AI 研究增强

在 P0a / P0b 稳定后加入：

1. 主线识别。
2. 龙头排序。
3. ETF 估值。
4. 个股估值。
5. 自动复盘评分。
6. 更多宏观和海外联动。

## 自动化需求

自动化由 Codex 自动化能力调度执行；表中的 AI 默认指 Codex 负责生成、分析、解释和审核前 JSON 初稿。

| 任务 | 实现方式 | 频次 | P 阶段 | 说明 |
|---|---|---|---|---|
| 交易日判断 | 脚本 | 每日盘前 | P0a | 使用交易日历决定是否运行。 |
| 数据采集 | 脚本 | 盘后每日，盘中按需 | P0a | Tushare 拉取盘后数据，QMT 只读确认盘中价格和真实持仓。 |
| 数据校验 | 脚本 | 每次采集后 | P0a | 检查缺失、过期、冲突和来源异常。 |
| JSON 研究生成 | AI | 每次研究 | P0a | AI 只输出 JSON，不直接输出 Markdown。 |
| JSON 校验 | 脚本 | 每次研究后 | P0a | JSON 不合格不得入库。 |
| 安全检查 | 脚本 | 每次发布前 | P0a | 检查密钥、金额、股数、账户号、真实订单和边界违规。 |
| 报告渲染 | 脚本 | 按需 | P0a | 从 JSON / SQLite 渲染 Markdown、HTML 或 PDF，不新增事实。 |
| 标的池更新 | 脚本 + AI | 盘后每日 | P0a | 脚本导入真实持仓，AI 只能提议主线 ETF、龙头、防御和观察标的。 |
| 市场仓位研究 | 脚本 + AI | 每个交易日 | P0a | 脚本生成市场事实包，AI 输出结构化 JSON。 |
| ChatGPT 审核包 | 脚本 | 每个交易日 | P0a | 打包 JSON 快照、数据缺口、冲突项、安全扫描结果和待审核问题。 |
| 人工确认 | 人工 | 审核后 | P0a | ChatGPT reviewed 后人工确认，才允许 finalized。 |
| finalized 入库 | 脚本 + 人工确认 | 审核后 | P0a | 由审核通过且人工确认的 JSON 写入最终库。 |
| 历史曲线更新 | 脚本 | 每次入库后 | P0b | 更新仓位、估值、真实持仓、影子账户和指数对比。 |
| 影子账户交易 | 脚本 | 每个交易日审核后 | P0b | 只在标的池内按固定规则生成纸面模拟交易。 |
| 异常提醒 | 脚本 | 每次任务后 | P1 | 提醒数据源失败、QMT 未登录、估值大幅变化或偏离目标。 |
| 主线和龙头研究 | AI | 每周或触发式 | P0c / P2 | 基于事实包判断主线强弱、龙头状态和失效条件。 |
| ETF 和个股研究 | AI | 每周或触发式 | P0c / P2 | 标的池和 ResearchFirst 队列优先。 |
| 组合分析 | 脚本 + AI | 每个交易日轻量，每周完整 | P0b / P1 | 脚本计算比例和暴露，AI 输出结构化 JSON。 |

详细验收规则、状态流和失败处理见 `design/REQUIREMENTS.md`。
